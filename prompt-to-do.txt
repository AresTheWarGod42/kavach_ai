THREE-LAYER DETECTION ARCHITECTURE
This is the core of Kavach.ai. Every layer is modular, independently testable, and composable into one unified scoring pipeline. Never collapse the layers into a single model.

L1 — Fast Gate (target: < 5ms, in-path, synchronous)
Purpose: First-pass deterministic filter. Eliminates obvious fraud and obvious clean transactions before any ML is invoked.

Components:

Rule Engine implemented as an in-memory Python dict + Redis BITFIELD for velocity counters:
Hard blocklist lookup on sender_bank / receiver_bank combos flagged historically.
Amount ceiling: flag any single transaction above ₹2,00,000 as ESCALATE immediately.
Velocity rule: if txn_count_1m > 5 per user, flag ESCALATE.
Weekend + late-night combo (is_weekend = True AND hour_of_day between 01:00–04:00 AND amount > ₹10,000) → ESCALATE.
Sub-second inter-transaction time clustering: if two transactions from same device_type arrive < 1 second apart → BLOCK (bot detection).
XGBoost first-pass classifier trained on Dataset 1 with a very low decision threshold (0.2) to maximize recall at L1 — it is better to over-escalate than to miss fraud here.
L1 Output: One of PASS, BLOCK, or ESCALATE.

PASS → transaction is allowed through, logged for baseline.
BLOCK → immediate soft-block pending analyst review (never a hard block without review).
ESCALATE → passes to L2.
Only ESCALATE transactions reach L2. PASS and BLOCK do not invoke L2 or L3.

L2 — Ensemble Core (target: < 20ms, in-path, synchronous, runs on ESCALATE only)
Purpose: Probabilistic fraud scoring using an ensemble of three base learners fed into a stacking meta-learner.

Base Learners (all trained on Dataset 1):

XGBoost — primary gradient-boosted tree; use Platt scaling for calibrated probability output.
LightGBM — leaf-wise growth; acts as a diversity anchor; faster inference than XGBoost.
Isolation Forest — unsupervised anomaly scorer; anomaly score is independent of fraud labels — this is what catches novel patterns outside the training distribution.
Stacking Meta-Learner:

A L2-regularized Logistic Regression meta-learner receives: the three base learner output probabilities + velocity features (txn_count_1m, txn_count_5m, txn_count_1h, amount_sum_1h) as meta-features.
Decision threshold is set at the F2-maximizing operating point on the validation set (β=2, recall-weighted).
Export all models via ONNX Runtime for inference. Pre-load ONNX models into memory at FastAPI pod startup — zero per-request loading overhead.
Compute SHAP values for every prediction at or above the Medium tier threshold (score ≥ 0.30). Return SHAP as a structured array in the /score API response JSON.
L2 Output: P(fraud) ∈ [0,1] + confidence interval + SHAP feature contributions array.

L3 — Deep Behavioral Detector (target: < 2000ms, fully async, post-authorization)
Purpose: Deep learning + graph-based enrichment using Dataset 2's behavioral and biometric signals. L3 runs out-of-path — it never blocks a transaction in real time. Its output enriches the audit log and feeds the nightly retraining queue.

Components (both trained exclusively on Dataset 2):

LSTM Autoencoder — trained on the sequential behavioral features (session_duration, input_timing_consistency, keyboard_input_speed, app_switching_frequency, screen_active_time, input_pause_patterns, time_between_otp_generation_and_input, pin_entry_speed). High reconstruction error = behavioral anomaly. Anomaly threshold set at 95th percentile of reconstruction error on the training set.

GraphSAGE (via PyTorch Geometric) — construct a transaction graph where user_id and merchant_id are nodes, and transactions are directed edges weighted by amount. GraphSAGE encodes 2-hop neighborhood context into node embeddings. High graph centrality anomaly score (PageRank deviation, clustering coefficient outlier, betweenness centrality spike) indicates a mule account or laundering chain. This is the only layer that can detect coordinated mule ring fraud invisible to per-transaction classifiers.

L3 Output: Reconstruction error score + graph anomaly score. Both are written to the audit log, displayed in the analyst alert queue, and used to trigger the retraining queue. L3 results are also passed into the Flowise LLM narration agent as enrichment context.

DEFENSE TECHNIQUES — IMPLEMENT ALL OF THESE WITHOUT EXCEPTION
These are not optional. They are core to the Blue Team challenge and must all be present in the final codebase.

1. Graduated Four-Tier Response Framework
Never implement binary block-or-pass logic. Every transaction that receives a score must be routed through this framework:

Hard blocks are never applied without analyst confirmation. False positive abuse (triggering FP flags to block legitimate competitors) is contained by requiring human sign-off at the Critical tier.

2. Adversarial Model Hardening
FGSM (Fast Gradient Sign Method): Apply FGSM perturbations to tabular features during training of the L2 deep components. Perturb features by ε = 0.01 in the direction of the loss gradient, retrain on both original and perturbed samples. This makes the model robust against fraudsters who tune transaction parameters to stay just below detection thresholds.
Differential Privacy: Apply Google's DP-SGD library (opacus for PyTorch components) to the LSTM Autoencoder training with a target ε = 0.5. This prevents membership inference attacks and ensures compliance with India's DPDP Act 2023.
Score Calibration: Apply Platt scaling + isotonic regression to L2 ensemble outputs. Any score outside the expected calibration curve at inference time triggers a system audit flag written to the audit log.
SHAP Gaming Detection: Monitor SHAP feature importance stability across model versions. If the top-5 SHAP features for a stable test cohort shift unexpectedly between versions, trigger a model review alert before promoting the new model to champion.
3. Concept Drift Handling and Online Learning
River (Python): Implement a Hoeffding Tree + online logistic regression that continuously updates on incoming labeled transactions (analyst-confirmed labels from the alert queue) without full retraining. This is the always-on incremental learner.
ADWIN (Adaptive Windowing): Run ADWIN on the fraud rate time-series. When the detected distribution shift exceeds the threshold, push a retraining job to a queue (simulate with a Python queue for demo).
Nightly Retraining: A scheduled job (Airflow or APScheduler) retrains on a rolling 30-day window. Use MLflow to version models. New model candidates run in shadow mode alongside the champion model — compared on identical scoring requests before promotion.
Champion-Challenger promotion is gated by: F2 > 0.87, AUC-PR > 0.92, FPR < 0.5%, and SHAP consistency check passing.
4. API and Infrastructure Defense
Per-VPA/per-user rate limiting via Redis sliding window. Throttle escalation: after 10 requests/minute, auto-escalate the next request to High tier regardless of score.
JWT authentication for all microservice-to-microservice API calls. Every internal FastAPI route requires a valid JWT bearer token.
Meta-level anomaly detection on API call patterns: Track the distribution of /score request feature vectors. If an incoming request's feature vector is a clear outlier in request-space (i.e., someone is probing the model with crafted inputs), flag it and log to the audit trail.
Circuit breaker: If scoring service P99 latency exceeds 200ms (measured via Prometheus), fall back to L1-only rule-based decision automatically. Auto-recover and re-enable L2 when latency normalizes. Log all circuit breaker events.
5. Cryptographic Audit Log and Non-Repudiation
Append-only audit log stored in PostgreSQL. Every scoring decision must write: transaction_id, timestamp, L1_decision, L2_score, tier, shap_top5, analyst_action (if any), model_version.
Hash chain: Each audit record stores the SHA-256 hash of the previous record's content. This makes the log tamper-evident by construction — any modification to a past record breaks the chain.
Data provenance tags on all training records: source dataset, ingest timestamp, label origin, SMOTE/ADASYN generation flag. Store in PostgreSQL alongside model registry metadata in MLflow.
6. Privacy-Preserving Design
Feature engineering operates exclusively on derived/aggregated signals — never raw PII.
ip_address, device_id, location, url_referrer are dropped at ingestion and never stored in the feature store or audit log.
No raw PII ever enters an LLM prompt. All Flowise/n8n prompts use only SHAP values, tier labels, and anonymized aggregate statistics.

Agent 1 — Alert Narration Agent (Flowise RAG chain)

Triggered on every transaction scoring at tier ≥ 3 (High or Critical).

Agent 2 — Adaptive Rule Agent (n8n workflow)

Triggered when ADWIN detects a drift event OR when recall drops below 0.80 in rolling evaluation.

Agent 3 — Threat Intelligence Agent (n8n HTTP nodes)

Runs on a configurable polling interval (default: every 6 hours).
